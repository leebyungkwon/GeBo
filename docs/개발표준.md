개발가이드 표준 정의서
1. 개요 및 목적
본 가이드라인은 엔터프라이즈 환경에서의 개발 일관성을 유지하고, 코드 품질을 상향 평준화하는 기술 표준이다. 본 표준의 준수는 기술 부채를 최소화하고, 시스템의 확장성을 보장하며, 소프트웨어 수명 주기 전반에 걸쳐 유지보수 비용을 절감하는 핵심 동력이 된다. 모든 개발자는 본 문서를 숙지하고 엄격히 준수해야 한다.

2. 코딩 컨벤션 (Coding Conventions)
2.1 Java 및 Spring Boot 표준
최신 기술 스택인  Java 21 이상  및  Spring Boot 3.x/4.x  환경을 기반으로 하며, 'Google Java Style Guide'를 핵심 지침으로 채택한다.
•	명명 규칙 (Naming Conventions):
•	클래스 및 인터페이스:  PascalCase를 사용한다. (예: UserProfile)
•	메서드 및 변수:  camelCase를 사용한다. (예: getUserData)
•	상수:  CONSTANT_CASE를 사용하며, 반드시 static final로 선언한다. (예: MAX_RETRY_COUNT)
•	포맷팅 및 하드 규칙:
•	들여쓰기:  탭(Tab) 사용을 금하며, **2개의 공백(spaces)**으로 고정한다.
•	열 제한 (Column Limit):  가독성을 위해 **110자(Soft Limit)**를 권장하며, **120자(Hard Limit)**를 초과하는 줄은 반드시 논리적으로 줄 바꿈(Line-wrapping)을 적용해야 한다.
•	중괄호:  K&R 스타일을 따르며, 선언과 같은 줄에서 시작하고 닫는 중괄호는 구문이 시작된 수직 위치에 맞춘다.
•	
2.2 JavaScript 및 Next.js 표준
Next.js 프레임워크의 특성과 파일 시스템 기반 라우팅의 효율성을 위해 다음 규칙을 적용한다.
•	파일명:  모든 파일 및 폴더명은  kebab-case 를 사용하여 URL 친화적이고 시스템 간 호환성을 확보한다. (예: user-profile.tsx)
•	리액트 컴포넌트:  파일 내부의 컴포넌트 명칭은 반드시  PascalCase 를 사용한다. (예: user-profile.tsx 내의 UserProfile)
•	변수 및 프로퍼티:  기본적으로  camelCase 를 사용하며, 상수인 경우에만  CONSTANT_CASE 를 허용한다.

2.3 자동화 도구 및 린팅 (Linting & Formatting)
도구에 의한 강제성을 부여하여 코드 리뷰 시 스타일 논쟁을 원천 차단한다.
•	도구 구성:  Java는  Checkstyle , JavaScript/TypeScript는  ESLint 와  Prettier 를 필수적으로 적용한다.
•	Git Hook 자동화:   Husky 와  lint-staged 를 이용하여 pre-commit 단계에서 린트 및 포맷팅 검사를 통과하지 못한 코드는 커밋할 수 없도록 강제한다.
•	에디터 설정:  IDE의  'Format on Save'  기능을 활성화하여 저장 시 즉각적으로 표준 포맷팅이 적용되도록 설정해야 한다.

3. 디렉토리 및 패키지 구조 (Directory & Package Structure)
3.1 Backend: Spring Boot 구조
Spring Boot 3.4.0 이상의 구조적 로깅 및 Jakarta EE 9+ 환경을 반영한다.
1.	소스 관리:  src/main/java에는 비즈니스 로직을, src/main/resources에는 설정 파일을 배치한다.
2.	로깅 설정 우선순위:  Spring Boot의 설정 탐색 순서를 엄격히 준수한다.
3.	logback-test.xml (테스트 환경 최우선)
4.	logback-spring.xml (Spring 프로파일 활용 권장)
5.	logback.xml (기본 설정)
6.	테스트 관리:  src/test/resources에 테스트 전용 설정 및 로깅 설정을 배치하여 운영 환경과의 간섭을 차단한다.
7.	
3.2 Frontend: Next.js App Router 구조
App Router의 파일 컨벤션과 논리적 구조화를 통해 유지보수성을 확보한다.
•	기능 기반 폴더링 (Feature-based folders):  관련 컴포넌트, 서비스, 훅을 기능 단위(예: features/auth)로 그룹화하여 관리한다.
•	Route Groups:  URL 구조에 영향을 주지 않는 논리적 분리를 위해  괄호  폴더명(예: (auth))을 사용한다.
•	주의:  서로 다른  Root Layout  간의 네비게이션 시 클라이언트 사이드 라우팅이 아닌 **전체 페이지 로드(Full page load)**가 발생하므로 성능 영향을 고려하여 설계해야 한다.
•	특수 파일 규칙:  각 라우트 세그먼트에서 page.js, layout.js, error.js, loading.js, not-found.js를 용도에 맞게 사용한다.

4. 형상 관리 및 브랜치 전략 (Version Control Strategy)
4.1 전략 비교 및 선택 기준
프로젝트의 성격에 따라 다음 두 전략 중 하나를 선택하여 운영한다.| 항목 | Git Flow | GitHub Flow || ------ | ------ | ------ || 복잡도 | 높음 (엄격한 5계층 구조) | 낮음 (단순 직선 구조) || 주요 브랜치 | Main, Develop, Feature, Release, Hotfix | Main, Feature || 적합한 환경 | 대규모 팀, 정기 출시 제품 (Spring Boot 4.x 기반 엔터프라이즈) | 소규모 팀, 지속적 배포 (SaaS, Next.js 프로젝트) || 핵심 원칙 | 배포 준비 완료된 코드는 Release 브랜치로 통합 | Main 브랜치는 항상 배포 가능한 상태를 유지 |
4.2 워크플로우 및 커밋 규칙
•	Pull Request (PR):  모든 코드는 반드시 1인 이상의 코드 리뷰를 거쳐 PR을 통해 병합되어야 한다.
•	검증 자동화:  모든 커밋은 Husky를 통해 사전 정의된 테스트 및 린트 검사를 통과해야 한다.
•	배포 가능성:  Main 브랜치로 병합되는 모든 코드는 즉시 운영 환경에 배포 가능한 수준의 품질을 유지해야 한다.

5. 로깅 및 가시성 표준 (Logging & Observability)
5.1 로깅 및 분산 추적 전략
분산 환경에서 시스템의 동작을 완벽히 이해하기 위해 단순 로깅을 넘어선 가시성(Observability) 전략을 채택한다.
•	프레임워크:  SLF4J 인터페이스와 Logback 구현체를 기본으로 사용한다.
•	성능 최적화:  I/O 성능 저하 방지를 위해  Async Appender 를 필수로 적용하며, 대규모 트래픽 시 backpressure 처리를 위해 OpenTelemetry의 배칭(Batching) 메커니즘을 활용한다.
•	분산 추적 (Distributed Tracing):   OpenTelemetry(OTel)  자동 계측(Auto-instrumentation)을 도입하여 모든 로그에 trace_id와 span_id를 삽입한다. 이는 Last9이나 Logstash와 같은 플랫폼에서 분산된 로그를 하나의 요청 단위로 연결하는 핵심 식별자가 된다.
•	로그 포맷:  분산 환경의 검색 용이성을 위해 운영 환경에서는 반드시  JSON 포맷 으로 출력한다.

5.2 전역 에러 핸들링 및 보안
•	Backend:  @ControllerAdvice를 통해 시스템 예외를 전역적으로 캡처하고, 클라이언트에게는 민감 정보가 제외된 표준 에러 응답 구조를 반환한다.
•	Frontend:  Next.js의  Error Boundaries 를 세그먼트 단위로 설정하여 에러 전이를 방지한다.
•	PII 보안 지침:  로그 적재 전(Pre-ingestion), Logback의 패턴 레이아웃이나 가시성 플랫폼의 제어 플레인을 통해  민감 정보(비밀번호, 카드번호, PII) 마스킹 처리 를 반드시 수행해야 한다.

6. 문서화 표준 (Documentation Standards)
6.1 코드 레벨 문서화 (TSDoc / JSDoc)
코드의 의도와 제약사항을 명확히 하기 위해 다음 태그를 적극 활용한다.
•	@param, @returns: 필수 인터페이스 설명
•	@deprecated: 폐기 예정인 API 명시
•	@remarks : 상세한 비즈니스 로직 및 부가 설명 기입
•	@beta : 실험적인 기능 또는 초기 단계의 API임을 경고
