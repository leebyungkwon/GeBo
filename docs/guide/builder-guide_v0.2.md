## 🛑 1. 시스템 존재 목적과 철학 (Core Philosophy)

이 **Builder 시스템을 구축하는 근본적인 목적**은 단순히 운영자가 화면을 직관적으로 만들기 위한 '찍어내기 도구'를 마련하는 것에 그치지 않습니다.
가장 중요하고 변하지 않는 설계 철학은 **"단일 진실 공급원(Single Source of Truth, SSOT) 구현과 시스템의 완벽한 상호 동기화"**를 달성하는 것입니다.

1. **무한한 재사용성:** 화면의 레이아웃, 조각(컴포넌트), 필드 하나라도 파편화되어 독자적으로 생성되지 않아야 합니다. 
2. **유지보수의 압도적 효율화 (One Update, Everywhere):** 코어가 되는 공통 컴포넌트 모듈 하나만 수정하면, 과거에 생성된 수천 개의 페이지와 미래에 만들어질 페이지 모두에 **추가적인 작업 없이 즉각적으로 완벽에 가깝게 반영**되어야 합니다.
3. **사용성과 결과물의 100% 일치(WYSIWYG):** 운영자가 어떠한 빌더 페이지(`quick-list`, `quick-detail`, `widget` 등) 화면에서 구성을 하더라도, 입력 화면(Input)과 실제로 가동되는 운영 화면(Output)이 시각적·기능적으로 완벽한 일치(Pixel Perfect)를 이루어야 합니다.

## 🎯 2. 공통화 핵심 설계 원칙 (Input & Output 통합)

이 시스템의 심장은 **레이아웃(Layout) / 콘텐츠(Content) / 필드(Field)** 컴포넌트들이 **"무조건 공통 컴포넌트 내부에서만 처리되어야 한다"**는 무관용 원칙입니다.

### 2.1. 어떠한 경우에도 공통 컴포넌트 파이프라인 탑승 원칙
화면을 구성하는 입력 빌더(Input)가 아무리 각기 다른 파일 경로(`quick-list`, `widget` 등)를 갖고 시작하더라도, 결국 이들이 조립해 내는 부품은 동일한 코어를 가져다 써야 합니다.
또한, 이를 그려주는(렌더링하는) 화면 역시 어떠한 예외 도피처 없이 통합된 Output 렌더러 블록을 사용해야 합니다.

* **입력 조립부 (Builder Components):** `SpaceBuilder`, `FormBuilder`, `SearchWidgetBuilder` 등
* **출력 렌더링부 (Renderer Components):** `SpaceRenderer`, `FormRenderer`, `SearchRenderer` 등
* **통합 분배 (Dispatcher):** 모든 렌더링 요소는 `WidgetRenderer` 또는 `CommonBuilderDispatcher` 라는 단일 허브를 통해 분기되어, 파라미터가 유실되는 일 없이 일관적으로 전달되어야 합니다.

### 2.2. 인라인 코딩(Inline Coding) 및 땜질식 개별 처리 엄격 금지
각 페이지(`page.tsx`)별로 특정 구조에서만 발생하는 문제(예: 레이아웃 여백 어긋남, 배치 오류 등)를 해결하기 위해 **해당 파일 내에 직접 `<div className="..." style={{...}}>` 와 같이 하드코딩하거나 예외 처리 코드를 삽입하는 것을 엄격히 금지**합니다.
어떤 환경(페이지)에서든 문제가 발생했다면, 그것은 해당 페이지의 문제가 아니라 **해당 공통 컴포넌트가 모든 제약사항(Props, Container Size)을 포용하지 못한 설계의 결함**으로 간주해야 합니다. 
문제 해결은 반드시 **공통 컴포넌트 내부(예: `SpaceRenderer`나 `WidgetRenderer`)의 규칙을 개선**하는 방향으로 이루어져야 합니다. 단 하나의 코드가 수정되더라도 전체 시스템의 통일성이 강건해지도록 조치해야 합니다.

## 🚀 3. 현재 진행 중인 고도화 내용 (v0.2 진척 사항)

우리는 기존에 부분적으로 파편화되어 동작하던(겉모양만 같고 구현 세부는 달랐던) `quick-list`, `quick-detail`, `widget` 빌더들을 **완벽히 통합된 하나의 공통 컴포넌트 아키텍처**로 엮어내는 작업을 진행하고 있습니다.

1. **설정 UI 모듈의 완벽한 100% 공통화**
   * 빌더 페이지마다 제각각 짜여 있던 **좌측 설정 패널의 뼈대 로직** (크기 설정 패널, 레이어 팝업 타입 지정, 위젯 헤더)을 완전히 분리했습니다.
   * 이제 `SizeSettingPanel`, `LayerPopupSettingPanel`, `ContentRowHeader` 등의 모듈화된 공통 컴포넌트만을 import 하여 조립합니다.

2. **렌더러(Output)의 정밀 격자 동기화 (Pixel Perfect Alignment)**
   * (진행 중 현황) 기존 `SpaceRenderer` 등에서 발생하던 문제—**각 빌더에서 넘겨주는 크기 규격(`contentColSpan`)이 통일되지 않아, 특정 빌더에서만 물리적인 수학 격자선에 버튼/배경이 미세하게 어긋나는 현상**—를 원천 수정 중입니다.
   * 단순히 `%` 너비로 밀어버리는 Flexbox 배치 구조를 삭제하고, 부모 요소에서 물려준 12칸 레이아웃 제약 사항(Props)을 렌더러가 온전히 흡수해 그 안에서 **CSS Grid 수학 공식대로 정확히 Snap(착상)** 되도록 코어를 변경하고 있습니다.
   * `quick-list`에서 누락되었던 렌더러 매개변수(`maxColSpan`, `contentColSpan`)들을 타 빌더와 완전 동일하게 주입되도록 맞춤으로써, 어떤 페이지든 똑같은 데이터를 투입하면 "반드시" 똑같은 형태가 나오도록 보정되었습니다.

3. **신규 생성 Item의 논리적 Default 값 통일화**
   * (진행 중 현황) Quick-List처럼 버튼만 등록 가능한 영역에서, 기존 시스템은 새 버튼에 무조건 최대 너비(`maxColSpan` = 12)를 강제 할당하여 레이아웃 인식의 이질감을 만들었습니다. 
   * 이를 해결하기 위해 개별 컴포넌트(`action-button`)의 논리적 기본 크기를 1로 제한하거나, 각 빌더 컨텍스트에 맞게 안전한 기본 크기가 할당되도록 `SpaceBuilder` 내부 코어 생성 함수를 리팩토링했습니다.

## 📂 4. 시스템 멘탈 모델 (데이터와 렌더링의 흐름)

빌더에서 화면은 결코 "페이지 파일 한 장"으로 만들어지는 것이 아니라, **블록들의 논리적 조합 데이터(JSON)** 와 **이 데이터를 읽어 뿌리는 렌더러 엔진**의 상호작용입니다.

```text
┌─────────────────────────────────┐
│     입력/수정 (Builder 3형제)      │
│ (widget, quick-list, quick-detail)│
└──────────┬───────────────────────┘
           │ [동일한 룰의 Props 설정]
           ▼
┌───────────────────────────┐   저장    ┌─────────────────┐
│         JSON Data         │ ───────▶ │   page_template │ (DB)
│ { widget: { align, items }│           └─────────────────┘
└──────────┬────────────────┘
           │ 
           ▼
┌────────────────────────────────┐
│   통합 렌더러 엔진 (Renderer)    │
│ (WidgetRenderer -> SpaceRenderer)│
└──────────┬─────────────────────┘
           │
           ▼
   [최종적으로 화면에 완벽 일치하여 표출]
```

**시스템이 요구하는 사고방식:**
* "내 빌더 화면에서 1px 엇나가는데 이 페이지에만 style을 추가해 패딩을 줄까?" ❌ 절대 금지
* "내 빌더가 렌더러 컴포넌트를 호출할 때 다른 빌더들과 다르게 취급받고 있는 데이터 파라미터 격차가 있는가? 컴포넌트의 CSS 분기 조건 설계가 허술한 것은 아닌가?" ⭕ 권장 작동 방식

Builder 시스템은 단 하나의 부품이 나머지 전체 생태계의 품질을 책임진다는 이 핵심 철학을 바탕으로 개발/유지보수되어야 합니다.
