데이터베이스 설계 표준 및 네이밍 규칙 가이드라인
1. 공통 네이밍 규칙 (General Naming Conventions) 모든 데이터베이스 객체에 적용되는 범용 규칙입니다. 예외 없는 준수를 원칙으로 합니다.
구분	규칙	설명 및 예시
기본 형식	소문자 및 snake_case	모든 식별자는 소문자만 사용하며 단어 사이는 언더스코어(_)로 구분합니다. (Good: user_profile / Bad: UserProfile)
길이 및 문자	최대 63자 제한	영문 소문자, 숫자, 언더스코어만 허용합니다. 숫자로 시작하거나 언더스코어로 끝날 수 없습니다.
금지 사항	예약어 및 따옴표 금지	SQL 예약어 사용을 금지하며, 식별자 정의 시 따옴표(")를 사용하지 마십시오. 따옴표 사용 시 호출 때마다 따옴표가 강제됩니다.
단어 선택	의미 있는 영어 단어	모호한 약어나 한글 병음 대신 완전한 단어를 사용하십시오. 단, i18n, l10n처럼 통용되는 약어는 허용합니다.
역할(Role)	용도별 접두사 부여	애플리케이션용: dbuser_<biz>, 권한 그룹: dbrole_<read/write>, 개인 계정: dbp_<ldap_id> 형식을 사용하십시오.
3. 객체별 상세 네이밍 규칙
3.1 테이블(Table) 및 뷰(View)
테이블 이름은 반드시 **단수형(Singular)**을 사용하십시오. 복수형(People, Persons 등)은 불필요한 언어적 모호함을 초래하며, 프로그래밍 언어의 클래스 매핑 시 혼선을 줍니다.
객체 유형	가이드라인	예시
일반 테이블	접두사(TB_) 사용을 엄격히 금지합니다.	user, organization
뷰 (View)	v_ 접두사를 사용하십시오.	v_active_member
구체화된 뷰	mv_ 접두사를 사용하십시오.	mv_sales_summary
임시 테이블	tmp_ 접두사를 사용하십시오.	tmp_migration_data
예약 스키마	public, monitor, dba, trash, timescaledb, citus, repack, cron은 사용을 금지합니다.	-
3.2 컬럼 (Column)
시스템 예약 컬럼명(oid, xmin, xmax, cmin, cmax, ctid)은 사용이 불가합니다.
성격	명명 규칙	예시
기본키 (PK)	id를 사용하십시오.	id
외래키 (FK)	참조테이블_id 형식을 준수하십시오.	user_id, org_id
날짜 및 시간	_date 또는 _time 접미사를 사용하십시오.	created_time, updated_time
불리언	is_ 또는 has_ 접두사로 상태를 명시하십시오.	is_active, has_permission
3.3 인덱스(Index) 및 제약 조건(Constraint)
시스템 생성 이름에 의존하지 말고, 아래 규칙에 따라 명시적으로 이름을 지정하십시오.
유형	접미사 규칙	명명 예시
기본키	_pkey	user_pkey
유니크 키	_key (Alternate Key 의미로 _ak도 가능)	user_email_key
일반 인덱스	_idx (컬럼 순서 포함 권장)	user_name_age_idx
외래키	_fk	order_user_id_fk
체크 제약	_ck	user_age_ck
배제 제약	_excl	room_reservation_excl
4. 데이터베이스 설계 표준 (Design Standards)
4.1 무결성 및 제약 조건 설계
데이터는 영구적이며 애플리케이션 로직은 변하기 쉽습니다. 무결성은 DB 레벨에서 강제해야 합니다.
•	Foreign Keys (FK): 고립된(Orphaned) 데이터를 방지하기 위해 반드시 설정하십시오. 삭제 시 액션(SET NULL, CASCADE 등)을 명확히 정의하십시오.
•	Unique Constraints: 비즈니스 로직의 중복 확인에 의존하지 말고 DB 레벨에서 중복을 원천 차단하십시오.
•	NOT NULL: 필수 데이터는 반드시 NOT NULL을 명시하여 애플리케이션의 방어적 프로그래밍 오버헤드를 줄이십시오.
4.2 테이블 구조 설계 원칙
•	기본키(PK) 선정: PostgreSQL 10 이상에서는 GENERATED ALWAYS AS IDENTITY 사용을 의무화합니다. 이는 시퀀스 동기화 오류를 방지합니다. (마이그레이션 예외 시에만 BY DEFAULT 허용)
•	Wide Table 지양: 컬럼이 15개를 초과하면 수직 분할을 검토하십시오. MVCC 메커니즘 특성상 컬럼이 많을수록 빈번한 업데이트 시 쓰기 증폭(Write Amplification)이 심화됩니다.
•	교차 테이블(Junction Table) 고도화: M:N 관계를 정의하는 교차 테이블은 단순히 두 ID의 결합을 넘어 관계의 메타데이터(role, joined_at, status 등)를 반드시 포함하여 관계 자체를 독립된 엔티티로 취급하십시오.
4.3 성능 및 확장성
•	파티셔닝(Partitioning): 단일 테이블이 10GB를 초과하거나 레코드가 1억 건 이상일 경우 파티셔닝을 설계하십시오. 개별 파티션의 크기는 1GB에서 64GB 사이로 유지하는 것이 가장 효율적입니다.
•	JSONB 활용 시점: 핵심 비즈니스 관계는 엄격한 컬럼 타입을 사용하고, 외부 API 로그나 가변적인 사용자 설정값 등 유연성이 절대적으로 필요한 경우에만 JSONB를 허용하십시오.
•	계산 로직 분리: 복잡한 연산은 애플리케이션 레이어에서 수행하십시오. DB는 데이터 필터링(PostGIS 등)에 집중해야 합니다.
5. 데이터 타입 선정 가이드
권장 데이터 타입	용도	주의사항
TEXT	문자열 데이터	CHAR(n) 사용 금지. UTF-8 환경에서 CHAR(n)은 고정 폭 저장 방식이 아니며, 불필요한 패딩으로 성능만 저하시킵니다.
NUMERIC	금액, 정밀 숫자	MONEY 타입은 통계/계산 시 유연성이 부족하고 로케일 종속적이므로 사용하지 마십시오.
BIGINT	PK 및 정수형	향후 확장성을 고려하여 ID 컬럼은 처음부터 BIGINT를 사용하십시오.
TIMESTAMPTZ	날짜 및 시간	timestamp 대신 반드시 사용하십시오. 타임존 정보를 포함하여 전 세계 시간 계산의 오류를 방지합니다. (ISO-8601 준수)
UUID (v1/v7)	고유 식별자	v4 사용 지양. v4의 무작위성은 B-tree 인덱스 내 '로컬리티'를 파괴하여 성능을 급격히 저하시킵니다.
IDENTITY	기본키 자동 생성	SERIAL 대신 GENERATED ALWAYS AS IDENTITY를 사용하십시오.
6. SQL 작성 및 쿼리 규칙
Do (권장)	Don't (지양)
명시적 컬럼 지정 (SELECT id, name)	SELECT * 사용 금지. 컬럼 추가 시 오류를 유발하고 불필요한 트래픽을 발생시킵니다.
NOT EXISTS 사용. 쿼리 플래너가 안티 조인(Anti-join)으로 최적화하여 성능이 우수합니다.	NOT IN 사용 금지. NULL 포함 시 결과가 왜곡되며 O(N^2) 복잡도로 성능이 붕괴될 수 있습니다.
Prepared Statements 활용. 파싱 오버헤드를 줄이고 보안을 강화합니다.	하드코딩된 값으로 쿼리 실행.
존재 확인 시 EXISTS와 LIMIT 1 사용.	존재 확인을 위해 COUNT(*) 사용.
트랜잭션 최소화. 10분 이상 유휴(Idle) 상태인 트랜잭션은 강제 종료 대상입니다.	트랜잭션 내에서 외부 API 호출 등 지연 작업 수행.
7. 저장 프로시저 및 트리거 사용 지침
•	트리거(Trigger)와 규칙(Rule): Rules 시스템은 사용을 엄격히 금지합니다. 이는 쿼리 재작성기로 동작하여 예기치 못한 결과를 초래합니다. 복잡한 로직은 Triggers를 사용하되, 디버깅의 어려움을 고려하여 신중히 도입하십시오.
•	저장 프로시저: 트랜잭션 캡슐화, 네트워크 라운드트립 감소, 또는 보안상의 이유로 SECURITY DEFINER가 필요한 시나리오에 한해 활용하십시오.
